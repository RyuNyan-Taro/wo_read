## 1. Goal
成長記録ページ内で、朝の準備完了時刻を1日1回だけSupabaseへ保存し、過去平均との早い/遅い判定と履歴振り返りを表示できるようにする。

## 2. Approach
既存アプリは `lib/main.dart:8-16` でSupabaseを初期化済みで、成長記録は `lib/record/record.dart:19-54` のStatefulWidgetがサービスからデータを取得して表示しているため、この構造に合わせて朝準備専用の model / service / use_case / screen を追加する。導線はユーザー回答どおり「成長記録内」とし、`lib/common/app_shell.dart:27-34` のBottomNavigationは増やさず、`RecordBody` の上部に朝準備セクションを差し込む。比較はユーザー回答どおり「平均比」とし、直近過去7件の平均 `ready_minutes` と今日の `ready_minutes` を比較して、早い/遅い/同じ/比較不可を判定する。

## 4. DB Design

### New Table: `public.morning_ready_records`
目的: 1日1回の朝準備完了時刻を保存し、日付単位の一意性と平均比較を安定して扱う。

```sql
create table public.morning_ready_records (
  id bigint generated by default as identity primary key,
  record_date date not null,
  ready_at timestamptz not null,
  ready_minutes integer not null check (ready_minutes between 0 and 1439),
  created_at timestamptz not null default now()
);

create unique index morning_ready_records_record_date_key
  on public.morning_ready_records (record_date);

create index morning_ready_records_record_date_desc_idx
  on public.morning_ready_records (record_date desc);
```

### Column Details
- `id`: Flutter側のモデル識別子。
- `record_date`: 端末ローカル日付を `yyyy-MM-dd` として保存する。1日1回制御の基準。
- `ready_at`: 実際にタップした日時。将来、秒単位の監査やタイムゾーン確認に使える。
- `ready_minutes`: `hour * 60 + minute`。過去平均比較をDB timezoneに依存せず実装するために保存する。
- `created_at`: Supabase側の作成時刻。

### Access Policy
既存の `glow_record` は `lib/record/service/record_service.dart:35-40` でanon clientからinsertされており、アプリ内認証は `lib/main.dart:11-14` のSupabase初期化以外に見当たらない。そのため初期実装は既存テーブルの公開/匿名アクセス運用に合わせる。Supabase側でRLSを有効にする場合は、このアプリに認証が入るまでは `anon` の select/insert を許可するpolicyが必要。

## 8. Risks & Mitigations
1. Timezone mismatch: `ready_at` の `timestamptz` から日付を計算するとSupabase/Postgres timezoneに左右される可能性がある。対策として、1日1回判定はアプリ側で作った `record_date` と `ready_minutes` を使い、DB側の日付変換に依存しない。
2. Anonymous access policy mismatch: 既存サービスはanon keyでDBへ直接insertしているが、Supabase側で新規テーブルだけRLSが有効だと保存に失敗する。対策として、既存テーブルと同じアクセス方針に揃え、RLS有効時はselect/insert policyを明示的に作る。
3. Duplicate tap race: UIの無効化だけでは通信遅延時の二重insertを完全には防げない。対策として、DBの `record_date` unique制約を必須にし、UI側でも保存中はボタンを無効化する。
4. Average calculation ambiguity: 平均対象が多すぎると古い生活リズムに引っ張られる。対策として、仕様を「直近過去7件平均」に固定し、画面にも「過去7件平均」と表示する。
5. `RecordBody` loading state expansion: 成長記録と朝準備記録のどちらか一方の取得失敗で画面全体が表示できなくなる可能性がある。対策として、初期実装ではエラー時に空配列へフォールバックしてセクション内に失敗表示を出すか、SnackBarで通知して既存の成長記録表示を維持する実装にする。
